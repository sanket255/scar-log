Day 16
Errors Have Names
a dev diary by scar  /  episode 16 of ∞

Turns out exceptions aren't a mysterious force that crashes your program for unknowable reasons. They're just errors with names. Specific, descriptive, catchable names. I've been writing except Exception for two weeks like a person who treats every illness with the same medicine and wonders why the results are inconsistent.

errors are just vocabulary

ValueError — you gave it text, it expected a number. Data type mismatch. Your fault probably.

FileNotFoundError — the file doesn't exist. Either you typo'd the name or you forgot to create it. Also probably your fault.

IOError — the file exists but you can't read or write it. Permissions. Corruption. Something external. Maybe not your fault this time.

Exception — the catch-all. Anything that didn't match the ones above. Fine for hobby projects. Not enough for production.

1 / 3
what i built (small, precise)

A file reader with three different catches. File doesn't exist? One message. Can't read it? Different message. Something else entirely? Tell me what it was. Ten minutes. Thirty lines. Completely understood.

The whole point: you can handle each error type differently. A missing file needs a different response than a corrupted one. Catching everything under Exception is like a doctor diagnosing everything as "being unwell." Technically covers it. Completely useless in practice.
hobby code vs thoughtful code

Generic Exception works fine when you're the only one using your program and you roughly know what might go wrong. The moment someone else touches it — or the moment you touch it six months later with no memory of writing it — specific error names are the difference between "File not found" and "Error: [Errno 2] No such file or directory: 'contacts.txt'" which tells you nothing useful when you're panicking at 2AM. Which is when these things always happen. Name your errors. Handle them specifically. Sleep better.

the actual realization

Exceptions aren't mysterious. They're just errors that Python was polite enough to label for you.

Once you know the names, you can talk back. That's all error handling is — a conversation between your code and the things that are about to go wrong.

Now I know the vocabulary.
